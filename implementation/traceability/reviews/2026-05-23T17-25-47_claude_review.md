## Summary

The plan adds two config knobs to the generic agent-review runner — `REVIEW_PROVIDER=claude|none` and `REVIEW_MODE=standard|lite` — plus two Runiac profile prompts (`06_manual_review_checklist.md`, `07_review_contract_lite.md`), preserving `claude + standard` as the default. It is correctly inspect-only: no files modified, dry-run preserved as default, implementation kept as a separate user-approved step, and commits kept manual with explicit `git add` commands. The scope correctly stays inside `tools/agent-review/` and lists protected paths to leave untouched.

The conservative framing is mostly sound, but the plan understates several concrete runner-level gaps. The most important: the runner hardcodes the review artifact name as `_claude_review.md` (`new_review_output_file()`), and the decision prompt/pipeline text hardcode "Claude review" — so a `REVIEW_PROVIDER=none` run would write a non-Claude artifact under a Claude-named file and feed it into a decision step that still tells Codex to "evaluate Claude's feedback." More fundamentally, `REVIEW_PROVIDER=none` introduces a sanctioned no-automated-review path into a tool whose entire purpose is to enforce an independent review gate; that is the opposite of conservative and is premature given the pipeline was only just added. I recommend shipping `lite` only and deferring `none`.

## Missing Constraints

- **No human-completion gate for the manual checklist.** Nothing requires the `none`-mode checklist to actually be filled before the decision/implement steps consume it. A label alone does not prevent an empty checklist from flowing downstream.
- **Pipeline + `none` interaction is undefined.** `cmd_pipeline` auto-chains plan → review → decision. The plan says it will "branch on provider" but does not state whether `none` is permitted in `pipeline` at all. It should not be auto-chained.
- **`REVIEW_PROMPT` vs `REVIEW_PROMPT_LITE` precedence** is unspecified.
- **No coupling of mode to change risk.** No rule keeps `standard` mandatory when a plan touches non-negotiable Runiac rules.
- **Prompt numbering jumps from `04` to `06`/`07`** with no `05` and no explanation. Confirm this is intentional.
- The plan keeps checklist *content* in the profile prompt (good, consistent with the runner-stays-generic rule) but should state this explicitly so the runner doesn't accumulate Runiac rules.

## Unsafe Operations

- The plan itself is clean on file safety: read-only, `DRY_RUN=1` default preserved, `ensure_new_file`/no-overwrite behavior intact, manual commits with explicit `git add <file>`, and the unrelated untracked file excluded.
- The one genuine integrity hazard is the mislabeled artifact (Risk 2): writing a non-Claude manual checklist into `implementation/traceability/reviews/` under a `_claude_review.md` filename would corrupt the governance/traceability record. The labeling-text mitigation does not fix the filename, which is generated independently by `new_review_output_file()`.
- No destructive operations, broad rewrites, or generated-file churn introduced.

## PRD/PDD Consistency Issues

- This batch touches only `tools/agent-review/` tooling and asserts no Runiac app features, so there is no direct conflict with `PRD.md`, the frozen submitted PDD (`docs/submissions/pdd/`), or `docs/pdd/`. The plan correctly lists those as untouched, consistent with the "do not edit/regenerate the submitted PDF" policy.
- Indirect concern: the agent-review pipeline is itself the enforcement mechanism for PRD/PDD discipline. A sanctioned `none` (no automated review) path raises the chance that a future plan's PRD/submitted-PDD drift goes undetected. This is governance-adjacent rather than a content mismatch, but it is the most consequential consistency risk in the batch.
- No Flutter/Firebase convention conflicts — no Flutter, Firestore, or Cloud Functions surface is touched.

## Overengineering Concerns

- **Premature expansion.** `tools/agent-review/README.md` "Future" explicitly says to wait for 3–5 real planning tasks before deciding to externalize/expand, and the pipeline was only just added (recent commits). Introducing a 2×2 provider/mode matrix now precedes that evidence.
- **`none` is anti-conservative for a governance tool.** The efficiency goal is met by `lite` alone. `none` adds a review-bypass path, which contradicts the batch's own "conservative, compatibility-first" framing.
- **Redundant API surface.** The plan admits `none + lite` and `none + standard` are the same path (`REVIEW_MODE` ignored when provider is `none`), so `REVIEW_MODE` silently no-ops under `none`. Either reject that combination explicitly or document the no-op.

## Recommendations

1. **Ship `REVIEW_MODE=standard|lite` only this batch; defer `REVIEW_PROVIDER=none`** until real usage justifies it. Keep `claude + standard` as default. This delivers the efficiency win without weakening the review gate.
2. **If `none` is retained:** forbid it in `pipeline`, allow it only in the manual `review` subcommand, require explicit human completion before the decision step, add prompt `03` and the pipeline preview/summary text to scope so no "Claude" wording survives, and **add an explicit artifact-naming branch** (modify `new_review_output_file()`/pipeline) so nothing is written as `_claude_review.md`.
3. **Define `REVIEW_PROMPT` vs `REVIEW_PROMPT_LITE` precedence** and document it in the env example and README.
4. **Keep `standard` mandatory for plans touching non-negotiable Runiac rules** (XP/level/rank/leaderboard ownership, roles/entitlements, premium fairness); state this in the lite prompt as an escalation instruction.
5. **Verify the `--tools` flag** against the real `claude --help` before relying on lite's tool-restriction guarantee; this pre-existing assumption underpins the "lite must not loosen tools" claim.
6. **Add a regression assertion** that standard mode still requires `02_claude_review_plan.md` after the `require_common_paths` refactor, and run the planned dry-runs for every standard/lite/none path.
7. **At implementation time, re-run `git status --short`**, exclude *both* untracked traceability files, and use explicit `git add <file>` (never `git add .`) — as the plan's approval gate already specifies.
8. Maintain the existing manual-commit and implementation-approval gates (the plan does). The `chore(repo):` commit type is acceptable; `docs(agents):` is arguably more accurate for workflow-prompt changes, per the AGENTS.md commit-style guidance — a minor judgment call.

No implementation should proceed until the user explicitly approves a revised plan; per my reviewer role I will not make these changes.
