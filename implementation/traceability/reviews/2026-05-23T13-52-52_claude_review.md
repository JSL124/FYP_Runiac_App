## Overengineering Concerns

- The `profiles/` tree is premature for a single consumer. The runner **already** overrides every prompt and output path via env vars and `AGENT_REVIEW_CONFIG` (lines 13–28), so a second project could be supported today without moving files. Introducing a profile layer + transition fallback + 3-batch rollout duplicates an existing capability.
- This contradicts the project's own guidance: `README.md` says "After 3–5 real planning tasks, review whether the runner logic is stable enough to externalize" and "Do not create a separate repo/submodule/package/Actions yet." Traceability shows only ~2 cycles so far (2026-05-23); the plan itself calls the workflow "newly tested."
- Leaner alternative: keep the current single layout; if/when a second project appears, point its env file at its own prompt paths, then reassess the restructure after the README's 3–5-run threshold.
